Discussion (vitejs/vite #22767): "How to configure Oxc minifier to Drop Console in Vite 8?"

> Oxc has an option to remove console calls in its minifier, but it's
> behind an option. There's `build.minify: 'oxc'` but no way to specify
> options for the minifier in `build` [...] alternatively
> `build.rolldownOptions.output.minify.compress.dropConsole: true`
> without `build.minify: 'oxc'` works.
> Should I do both, or is the `rolldownOptions` enough? and should there
> be a property to configure oxc minifier (like `terserOptions`) in build?

Repository context (curated from the accepted answer on the same thread):

- Tested on vite@8.1.0 in lib mode: `build.minify: 'oxc'` on its own
  leaves all console.* calls in; no compress config is plumbed through
  that path.
- `build.rolldownOptions.output.minify.compress.dropConsole: true` is
  what actually strips them, and works without also setting
  `build.minify: 'oxc'` (oxc is already the minifier under the hood).
- Setting both gave byte-identical output to rolldownOptions alone.
- There is currently no terserOptions-style key for the oxc minifier in
  `build`; routing compress options through
  `build.rolldownOptions.output.minify` is the way for now.
  `dropConsole` is a real rolldown CompressOptions field.

Answer the discussion.

output

Use only:

```js
export default {
  build: {
    rolldownOptions: {
      output: {
        minify: {
          compress: {
            dropConsole: true,
          },
        },
      },
    },
  },
}
```

No need for `build.minify: 'oxc'`; Oxc is already used underneath. On Vite 8.1.0, setting both produced byte-identical output.

There is currently no `terserOptions`-style Oxc configuration key. For now, pass `dropConsole`—a valid Rolldown `CompressOptions` field—through `build.rolldownOptions.output.minify`.

Discussion (fastapi/fastapi #15989): "Invalid contact/license_info URL
fails lazily at first /openapi.json request, not at app construction"

> FastAPI accepts `contact={"url": ...}` (and `license_info`) at app
> construction time without validating the URL. The value is only
> validated the first time `/openapi.json` is requested — so a mistake
> made when constructing `FastAPI(...)` doesn't surface until an
> unrelated request later, as a 500 buried in Starlette internals.
> Is there a way to make this fail at startup instead?

Repository context (curated from the accepted answer on the same thread):

- contact.url is only validated when the OpenAPI model is built, at
  `OpenAPI(**output)` at the end of openapi/utils.py. FastAPI() stores
  it as a raw dict at construction and builds the schema lazily on the
  first /openapi.json or /docs hit, then caches it.
- To fail at startup: call `app.openapi()` once right after constructing
  the app — same ValidationError on info.contact.url, but at the point
  where the typo actually is. Checked on 0.115.0, unchanged on master.

Answer the discussion.

output

Yes. Force OpenAPI generation immediately after constructing the app:

```python
app = FastAPI(
    contact={"url": "..."},
    license_info={"url": "..."},
)

app.openapi()
```

`FastAPI()` stores these as raw dictionaries. URL validation occurs later when `OpenAPI(**output)` builds the schema; `/openapi.json` and `/docs` trigger this lazily and cache the result. Calling `app.openapi()` directly makes the same `ValidationError` surface during startup instead.

Discussion (fastapi/fastapi #15973): "Behavioral difference between

> I'm wrapping a sync context manager with contextmanager_in_threadpool.
> When __enter__ raises, I expect __exit__ to NOT be called, per PEP 343
> (the context was never entered). Instead, __exit__ IS called.
> Is this a real deviation, and is a fix known?

Repository context (curated from the accepted answer on the same thread):

- Confirmed: moving `__enter__` out of the `try` fixes the first issue
  cleanly (and stops `__exit__` swallowing the `__enter__` exception
  when it returned True).
- Two further PEP 343 deviations remain: the guard catches `Exception`
  but not `BaseException` (so KeyboardInterrupt / SystemExit /
  CancelledError raised in the body never reach `__exit__`; the guard
  needs to be `except BaseException`), and a real anyio task
  cancellation never reaches `__exit__` because the
  `anyio.to_thread.run_sync` call is itself a cancellation checkpoint.
- The normal-exit branch has the same cancellation window; the fix is
  to shield every `__exit__` thread call.

Answer the discussion.

output

Yes. It is a real PEP 343 deviation.

Moving `__enter__` outside the `try` prevents `__exit__` from running—or swallowing the exception—when `__enter__` fails.

Two related fixes are also needed:

- Catch `BaseException`, not only `Exception`, so body exceptions such as cancellation reach `__exit__`.
- Shield every threaded `__exit__` call from AnyIO cancellation, including normal exit.
